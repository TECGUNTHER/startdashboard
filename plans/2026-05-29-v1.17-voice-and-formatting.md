# Build Plan: StartDashboard v1.17 — Gemini chat voice input + formatted replies

**Date:** 2026-05-29
**Status:** Approved
**Plan author:** Design Agent (via Orchestrator)
**Approved by:** Tom (chose: mic input + formatted replies; NO text-to-speech this round)

## The Idea, In Plain English

Add a 🎤 button to the Gemini chat input so Tom can speak his question (transcribed into the box). And render Gemini's markdown replies with real formatting — bold, lists, code blocks, clickable links — instead of flat text with raw markdown characters.

## Context

Web Speech API (`webkitSpeechRecognition`/`SpeechRecognition`) gives free browser speech-to-text (Safari + Chrome + iOS Safari, over HTTPS = GitHub Pages). The chat's `makeChatBubble(role, text, isHtml)` (~line 5394) already has an `isHtml` hook, so rendering markdown into model bubbles is a clean addition. No key, no backend, no library.

## What It Will Do
- **Mic:** 🎤 button in `.chat-input-row` (next to Send). Click → listening state (accent/pulse) → speech transcribes into `#chat-input` → click again / pause stops → review + Send. Hidden if the browser lacks speech support. Mic-permission denial shows a friendly note. Works on iOS Safari.
- **Formatted replies:** Gemini messages render bold, italic, inline code, fenced code blocks, bullet + numbered lists, headers, and http(s) links (new tab, rel=noopener). Done via a SAFE vanilla `renderMarkdownSafe(text)` (escape-first, whitelist tags only — no arbitrary HTML/scripts). User/note/thinking bubbles stay plain. Chat CSS styles the markdown elements for light + dark.

## What It Will NOT Do
- No text-to-speech / read-aloud (future option).
- No rich-text input editor (textarea stays plain — correct for AI input).
- Not a full CommonMark parser (safe practical subset; exotic markdown degrades to readable text).
- No new key/backend/library.
- No change to the Gemini request flow, model, history, 503 retry, or link-out.

## Approach
1. **Mic:** add `<button id="chat-mic">` to `.chat-input-row` (HTML ~2069) + CSS + `.listening` state. JS: hide if `!('webkitSpeechRecognition' in window || 'SpeechRecognition' in window)`; lazily create recognition (`interimResults=true`, `continuous=false`, browser lang); click toggles start/stop; `onresult` writes transcript to `#chat-input` + `autoGrowChatInput()` (~5646); `onend`/`onerror` reset button; `not-allowed`/`denied` → friendly note.
2. **Markdown:** `renderMarkdownSafe(text)` — escape `& < > "` (reuse `escapeHtml`), then ordered transforms: fenced code blocks first (protect contents), then headers, bold, italic, inline code, links `[t](url)` http(s) only, ul + ol lists, paragraphs/line breaks. Emit only whitelisted tags (strong, em, code, pre, ul/ol/li, h3, a, br, p). In `renderChatMessages` (~5364) and the model-reply append (~5580): for `role==='model'` use `makeChatBubble('model', renderMarkdownSafe(text), true)`. Leave user/note/thinking as-is.
3. **CSS** for `.chat-msg.model` markdown: code blocks (mono, scrollable, tinted), inline code, ul/ol spacing, links (accent), headers, paragraph spacing — theme tokens, light + dark.
