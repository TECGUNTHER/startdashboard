# Build Plan: StartDashboard v1.16 — Reliable card rearranging (smooth pointer-drag + landing highlight + move controls)

**Date:** 2026-05-29
**Status:** Approved
**Plan author:** Design Agent (via Orchestrator)
**Approved by:** Tom (chose: smooth drag + move controls; explicitly added: a clear highlight of where the card will lock in)

## Context

Rearranging widget/category cards is cumbersome and unreliable. Two diagnosed causes: (1) native HTML5 drag-and-drop is flaky — `drop` doesn't always register, so cards "don't move / hang"; (2) the v1.15 masonry switch left the before/after test using HORIZONTAL geometry (`(clientX-rect.left)>rect.width/2`) while masonry packs cards VERTICALLY in columns — wrong axis, so hovering the expected side does nothing. Fix: replace the native drag for widget + category cards with a vanilla pointer-drag engine that shows a clear highlighted landing slot, plus explicit move controls.

## What It Will Do
- **Smooth drag:** press the ⠿ grip → card lifts (scaled/translucent clone follows the pointer) and a **highlighted placeholder slot** (dashed accent box, card-sized) shows where it will land, tracking the cursor. Release → card locks into that slot; order saves + re-packs. Forgiving (nearest-slot, no precise half-hit needed). Pointer-events based → also works on touch (iPhone bonus).
- **Move controls:** every widget + category gets move earlier / later / front / end (operates on the order array — predictable regardless of masonry packing). Preferred placement: small move arrows near the grip (one-click), and/or entries in the widget ⚙ config + category ⋯ menu (Build Agent picks cleanest discoverable spot).
- Preserved: glow colors, gear configs, watchlists, masonry, backgrounds (incl. Starlink), light/dark, Gemini chat, bookmark add/move.

## What It Will NOT Do
- No change to the masonry layout itself (v1.15 stays).
- Does NOT rebuild bookmark drag or persona-tab/config-list drag (scope = widget + category cards). They can adopt the engine later.
- No external drag library (vanilla only).
- No per-card resize/width-spanning.

## Approach
1. Remove native `draggable`/`dragstart`/`dragover`/`dragleave`/`drop` reorder handlers + horizontal before/after geometry on widget cards and category cards. Keep order-array mutators.
2. Shared vanilla **pointer-drag engine** (widgets + categories):
   - `pointerdown` on grip → `setPointerCapture`; record dragged card + order key/index; create a `position:fixed` floating clone following the pointer; insert a `.drop-placeholder` (card-sized, highlighted) at the start position.
   - `pointermove` → nearest card by 2D center distance among visible cards in that container → move placeholder before/after it; reposition the clone to the pointer.
   - `pointerup` → final index from placeholder position → splice order array (`persona.widgetOrder` / `persona.categories`) → `setState` (persist + re-render) → remove clone/placeholder.
   - Cancel cleanly on `pointercancel` / Escape.
3. `.drop-placeholder` CSS — dashed accent box, soft accent fill, card-sized; clearly visible in light + dark (Tom's "landing highlight").
4. `moveCardInOrder(orderArray, key, direction)` for `earlier|later|front|end`; wire to per-card move arrows/menu items for widgets + categories; `setState` persists.
5. Apply to BOTH widget cards (`renderWidgets`/`buildWidgetCard`) and category cards (`renderCategory`). Trailing "+ New category" tile and "+ Add watchlist" affordance excluded as drag targets; layout stays correct.
