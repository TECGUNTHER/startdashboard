# Build Plan: StartDashboard v1.18 — Reliable rearranging (Reset button + arrows-primary + de-flickered drag)

**Date:** 2026-05-29
**Status:** Approved
**Plan author:** Design Agent (via Orchestrator)
**Approved by:** Tom (chose: Reset button + move arrows as the reliable primary + de-flicker the drag; keep masonry)

## Context

Card drag is unreliable because the v1.15 masonry (CSS multi-column) layout fights live drag. Confirmed in code (`initCardDrag` ~2979, `updatePlaceholder` ~3037): (1) BLINKING — `updatePlaceholder` calls `container.insertBefore(placeholder,…)` on every `pointermove`, re-packing the whole masonry each pixel; (2) "HIGHLIGHTS BUT DOESN'T STAY" — masonry packs by height, so the final drop re-packs and the card can land off from the preview. Decision: keep masonry; make rearranging reliable via Reset + move arrows (primary) + a de-flickered (approximate) drag.

## What It Will Do
- **Reset layout** (Settings, per-persona): restore widgets to default order + clean re-render so masonry re-packs tidily; categories re-pack on the same action. Toast "Layout reset." Non-destructive (order only).
- **Move arrows primary:** ▲▼ (earlier/later) shown always at low opacity (brighten on hover) on every widget + category; ADD "Move to front" / "Move to end" (helper `moveCardInOrder` already supports `front`/`end`) in widget ⚙ config + category ⋯ menu. Exact, predictable, persisted.
- **De-flickered drag:** remove the in-flow `.drop-placeholder` insertion during `pointermove`; instead toggle an overlay indicator class on the NEAREST card (`.drop-target-before`/`.drop-target-after`, accent edge via box-shadow/pseudo — no layout change → no reflow → no blink). Only update when target/side changes. On `pointerup`: compute target from nearest card + before/after, splice order via existing `onDrop`, single `setState`. Keep Escape/cancel. Drag is honestly approximate (lands near indicator); arrows + Reset are the precise/recovery tools.
- Preserved: masonry, glow colors, gear configs, watchlists, backgrounds (incl. Starlink), light/dark, Gemini chat (voice+formatting), bookmark add/move, gist persistence.

## What It Will NOT Do
- No JS-absolute-positioned tile rebuild (declined "premium smooth drag").
- No abandoning masonry (declined equal-height grid).
- No pixel-perfect drag promise — stops flicker + clean commit, but exact landing stays approximate in masonry.
- No change to Gemini chat / widget data / sync.

## Approach
1. **`initCardDrag` (~2979–3122):** drop the in-flow placeholder insertion; on move, find nearest card (existing nearest-by-center ~3045–3061) and toggle `.drop-target-before`/`.drop-target-after` on it (visual only, guarded against redundant updates); clone still follows cursor. On `pointerup`: target index from nearest card + before/after → existing `onDrop` → single re-render; clear indicator classes. Keep Escape/cancel.
2. **Move controls:** `.move-controls`/`.move-btn` visible at low opacity by default (brighten on hover); surface `moveCardInOrder(...,'front')`/`'end')` as "Move to front"/"Move to end" in widget ⚙ config + category ⋯ menu via `setState`.
3. **Reset:** "Reset layout" button in Settings (per-persona). On click: set `persona.widgetOrder` to the default order ensurePersonaSchema uses (builtins `clock,weather,todos,notes,crypto,markets` then `watchlist:<id>` keys), force clean re-render (categories re-pack as-is), `setState` persist; toast "Layout reset."
