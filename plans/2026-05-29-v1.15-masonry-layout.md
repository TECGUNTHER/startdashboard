# Build Plan: StartDashboard v1.15 — Masonry layout for widgets + categories

**Date:** 2026-05-29
**Status:** Approved
**Plan author:** Design Agent (via Orchestrator)
**Approved by:** Tom (chose: masonry/vertical-packing; applied to BOTH widgets and categories)

## The Idea, In Plain English

Make widget cards and category cards pack vertically like Pinterest tiles: each card is only as tall as its own content; a tall card (e.g. a long Watchlist) extends down while the cards below pack up beside the short ones, instead of every card in a row being forced to the tallest card's height. Reclaims wasted space, far less scrolling. Drag-reorder (⠿ grips) keeps working.

## Why

`.widgets-row` is `display:grid; grid-template-columns:repeat(4,1fr)` — grid rows force equal heights, so a tall Watchlist stretches Clock/To-do/Notes and pushes categories far down. `.categories-grid` (auto-fill grid) has the same issue. Switch both to CSS multi-column masonry.

## What It Will Do
- Widgets: natural card heights; tall card extends down; shorter neighbors stay compact; second-row cards pack up beside them. Responsive column-count 4 / 2 (≤1100px) / 1 (≤600px) — existing breakpoints.
- Categories: same vertical packing; short categories not stretched by tall neighbors. Responsive via `column-width: 280px` (matches current auto-fill feel).
- Drag-reorder via ⠿ grips still works for both; order persists to gist.
- All per-card features intact: glow colors, gear configs, watchlists, backgrounds (incl. Starlink), light/dark.

## What It Will NOT Do
- No internal-scroll height caps (rejected alternative).
- No per-card width spanning this round (future granularity option).
- No data-model/content/sync change — CSS layout + any drag-logic adjustment only.
- No experimental `grid-template-rows: masonry` — use well-supported CSS multi-column.

## Files to Be Changed
| File | What |
|------|------|
| `projects/personal-dashboard/index.html` | `.widgets-row`/`.widget` and `.categories-grid`/`.category` → CSS multi-column masonry; responsive breakpoints; verify/adjust drag-reorder drop logic for column flow |

## Approach
1. `.widgets-row`: drop `display:grid`+`gap`; use `column-count:4; column-gap:16px;`. Responsive: `column-count:2` ≤1100px, `column-count:1` ≤600px.
2. `.widget`: `break-inside:avoid; width:100%; margin-bottom:16px;`. Keep `min-height` + internal flex-column (break-inside:avoid works on flex containers; Build Agent resolves the display detail so cards don't split across columns).
3. `.categories-grid`: replace grid with `column-width:280px; column-gap:16px;`.
4. `.category`: `break-inside:avoid; width:100%; margin-bottom:16px;`.
5. **Preserve drag-reorder (risk area):** widgets (`persona.widgetOrder`) and categories (`persona.categories` order) reorder via ⠿ grips + position-aware HTML5 drag (top/bottom half of target → before/after in DOM order). DOM order drives multicol flow, so array reorder re-packs columns. Build Agent MUST verify reorder works for both after the switch and adjust drop-position geometry if needed. "+ New category" tile and "+ Add watchlist" affordance must still render/work as trailing items.
