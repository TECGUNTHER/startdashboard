# Build Plan: StartDashboard v1.13 — Gemini chat 503/overload resilience

**Date:** 2026-05-29
**Status:** Approved
**Plan author:** Design Agent (via Orchestrator), informed by Research Agent findings
**Approved by:** Tom (reported the bug; this is a reliability fix to restore intended chat behavior)

## Context

Tom asked the in-portal Gemini chat a question; the first answer worked, the follow-up returned: *"This model is currently experiencing high demand. Spikes in demand are usually temporary. Please try again later."* This is a Gemini **HTTP 503 "model overloaded"** response — a Google server-side capacity issue for that model tier, unrelated to the user's quota, account, or time of day (confirmed via research). The standard remedy is automatic retry (transient 503s usually clear within a second or two). v1.11's chat handles network/CORS and 429 errors but treats a 503 as a dead end and surfaces it immediately. This fixes that.

## What It Will Do
- **Auto-retry on 503 / "overloaded" / "high demand" responses** before showing any error: up to 2 retries with short backoff (~1.5s, then ~3s) plus small random jitter. Most transient 503s clear on the first retry.
- **During retries**, keep showing the "Gemini is thinking…" indicator (don't flash an error mid-retry).
- **If all retries fail**, show a clear, plain-English message: "Google's servers are briefly overloaded (this is on Google's side, not you). Try again in a moment." — NOT the raw API text — plus:
  - a one-click **Retry** button on that failed exchange (re-sends the last user message), and
  - the existing "Open full Gemini ↗" link-out remains available.
- **Keep existing handling** for 429 (rate-limit message) and network/CORS (link-out fallback) unchanged.

## What It Will NOT Do
- No exponential backoff with 30s+ waits (too slow for a chat; short retries fit the UX). Cap at 2 retries.
- No automatic model-switching/fallback (default model `gemini-2.5-flash` is already a relatively stable tier; adding model fallback is out of scope for this fix).
- No change to history, settings, or the link-out.

## Files to Be Changed

|------|------|
| `projects/personal-dashboard/index.html` | `sendChatMessage` error handling: detect 503/overloaded, auto-retry with backoff, clearer exhausted-retry message, add a Retry button affordance to failed exchanges |

## Approach
- In `sendChatMessage` (or a helper it calls), detect a 503 status OR a response whose error text contains "overloaded"/"high demand". On detection and if retries remain, wait the backoff (with jitter) and re-issue the same request, keeping the thinking indicator up. Cap at 2 retries.
- On exhaustion, render a `.note`-style bubble with the friendly message + a Retry button that re-runs the last user turn. Reuse existing chat bubble/CSS patterns and theme tokens.
- Reuse the existing fetch/error structure from v1.11; do not add dependencies or a backend.
