Last updated: 2026-05-29 · v1.14
01

Architecture Overview

How the app is structured and how the pieces fit together.

StartDashboard is a single HTML file. Everything — markup, styles, and behavior — lives in index.html (~5,380 lines as of v1.14). There is no build step, no framework, and no server. The page is served as a static asset (GitHub Pages in production; you can also double-click the file locally).

The app has three runtime layers:

  • State. A single state object holds everything: the GitHub token, gist ID, default persona, and Finnhub API key (all from localStorage on boot), plus the synced state.data object that mirrors the gist's JSON. (The old Twelve Data key is still read into state.twelveDataKey for backward-compat but is no longer used by any app logic as of v1.9.) All UI rendering reads from this object; all mutations go through setState() which re-renders and triggers a debounced save. The Settings save flow bypasses the debounce and awaits the save directly so fast reloads always see the latest state.
  • Sync. Reads/writes go to the GitHub REST API at api.github.com/gists/:id. Loading is a single GET; saving is a single PATCH. Background poll every 60s checks updated_at and surfaces a "stale" indicator if another browser updated. A cross-window localStorage message (sd_last_added) lets the bookmarklet popup notify the main dashboard about a fresh addition, surfacing a "Switch to X persona" banner when the add went somewhere the user isn't currently viewing.
  • Render. Vanilla DOM manipulation. The whole dashboard re-renders on persona switch. As of v1.10 the widget row is built dynamically (see below) rather than from static HTML. Per-widget render functions: clock (re-rendered every 30s), weather (fetched from Open-Meteo on render; as of v1.12 the daily request also asks for precipitation_probability_max, and each forecast day renders a condition icon via weatherCodeIcon(code) + short label and a chance-of-rain line), todos, sticky notes, crypto (CoinGecko, fetched on render), markets and watchlists (Finnhub as of v1.9, one /quote call per symbol fanned out via Promise.all, drawing from a shared 5-minute localStorage quote cache so only stale/missing symbols hit the network). The shared render function for stock widgets is renderStockWidget(symbols, bodyId, emptyMsg, refresh) (signature widened in v1.10 so it can serve any watchlist card, not just a single one); it's called by renderMarketsWidget and, per watchlist, by renderStockWidgetFor.

The bookmarklet adds bookmarks via a separate code path. When the URL contains ?add=<url>&title=<title>, the page renders only a compact "Pick a persona + category" overlay, writes synchronously to the gist, and closes its window. The main dashboard is never instantiated in that mode.

Per-persona widget visibility (persona.hiddenWidgets) controls which widget cards render. A persona can have zero widgets (the entire row collapses), or any subset of the available widgets. Each persona also has its own background, accent color, search engine config, and clock format (12hr vs 24hr).

Dynamic widget row + the watchlists array (v1.10). The widget row used to be static HTML. As of v1.10, renderWidgets empties #widgets-row and rebuilds every card from the persona's widgetOrder array (filtered by hiddenWidgets), via the helper buildWidgetCard. This was required to support both card reorder and multiple watchlist instances. Built-in widget keys are clock, weather, todos, notes, crypto, markets. Watchlists are no longer a single widget: persona.watchlists is an array of { id, title, symbols[] }, and each one renders as its own card under a composite key watchlist:<id>. That composite key is used everywhere a widget is referenced — widgetOrder, hiddenWidgets, widgetColors, the visibility toggles, and the config modal — so a watchlist participates in the same reorder/hide/color machinery as a built-in widget without special-casing. Each widget card header carries a ⠿ grip (reusing the existing .grip / .drop-before / .drop-after classes); moveWidget handles a horizontal, position-aware drop and rewrites widgetOrder. renderStockWidgetFor renders one watchlist card; addWatchlist / deleteWatchlist manage the array. Markets stays a singleton index-proxy card and is unchanged. (Implementation note: because the row is fully regenerated each render, the Notes textarea is rebuilt every time but keeps its notes-textarea id; closing a widget config now calls a full render() so title/color/symbol edits propagate everywhere at once.)

Per-card border/glow color (v1.10). Two storage locations: category.color (optional hex per category; null = inherit the persona accent) and persona.widgetColors (a map of widgetKey → hex; an absent entry = inherit accent). A colored card gets a colored border plus a soft outer glow (box-shadow) — subtle, no animation. applyCardColor(el, hex) applies the border/glow to a card element. The picker UI is built by buildCardColorControl (preset swatches + a native color picker + a “Default (theme)” reset) and is embedded in each widget's ⚙ config; for categories, openCategoryColorPopover opens the same control from the category's ... context menu. To-do and Notes, which previously had no config, gained a new color-only ⚙ modal.

Stock quotes (v1.9). fetchFinnhubQuotes(symbols) fans out one finnhub.io/api/v1/quote?symbol=X&token=KEY call per symbol via Promise.all and normalizes Finnhub's c (current), d (dollar change), dp (percent change) fields into the dashboard's { price, dollar, percent } shape. A per-symbol failure degrades to "—" rather than failing the whole widget. Results flow through a shared 5-minute localStorage quote cache (sd_quote_cache, TTL QUOTE_CACHE_MS = 5min, shape { SYM: { price, dollar, percent, t } }, managed by readQuoteCache() / writeQuoteCache()). Only stale or missing symbols hit the network, and the cache is shared across all tabs and reloads — this replaced the old per-tab in-memory _tdCache. If every symbol is rate-limited with nothing cached, fetchFinnhubQuotes throws an error tagged isRateLimit; renderStockWidget then shows "Rate limit reached — prices refresh shortly." and schedules one refresh 60 seconds later (no immediate retry — the v1.8 2-second retry made rate-limiting worse). The crypto widget (CoinGecko) is unchanged and keeps its v1.8 single retry; it has no shared cache.

Drag-to-reorder (v1.9). Every draggable element prepends a visible ⠿ grip handle (.grip, grab cursor, brightens on hover); a .drop-before / .drop-after accent line indicates the insertion point. Categories reorder via a grip in renderCategory. Bookmarks use position-aware dragover/drop (top vs bottom half of the target row) routed through moveBookmark(bkId, fromCatId, toCatId, beforeId, after), which handles both within-category and cross-category moves; a drop on empty category space appends. Config lists (markets/watchlist tickers, weather cities, clock zones in renderWidgetCfgList) use the shared helper makeCfgItemDraggable(li, index, reorderArray), which prepends a grip and reorders the persona's underlying array on a position-aware drop.

Ask Gemini chat panel (v1.11). A floating action button (FAB, bottom-right) opens a slide-over chat overlay. It is a global panel, not a per-persona widget — it sits outside the persona/widget machinery and is available from every persona. The FAB is shown only on the main dashboard (showDashboard) and hidden on the setup and add-bookmark screens (showSetup / showAddOverlay). The overlay (backdrop + panel) uses z-index: 95 — above the FAB (90) but deliberately below the existing modal overlays (100), so a Settings/config modal still layers on top. Enter sends, Shift+Enter inserts a newline, and Escape / a backdrop click / the ✕ button close it. Sending calls Gemini directly from the browser (no backend): sendChatMessage issues a POST to https://generativelanguage.googleapis.com/v1beta/models/<model>:generateContent with headers Content-Type and x-goog-api-key, and a body of { system_instruction: { parts:[{ text: GEMINI_SYSTEM_PROMPT }] }, contents: [ …last 10 turns as { role, parts:[{ text }] } ] }; it parses the reply from candidates[0].content.parts[0].text. The default model is gemini-2.5-flash, overridable via a Settings field. GEMINI_SYSTEM_PROMPT primes concise, search-style answers. Chat history is stored per-device in localStorage (sd_chat_history, an array of { role:'user'|'model', text }, trimmed to the last CHAT_HISTORY_CAP = 50 by saveChatHistory) and is not synced through the gist — deliberately, to keep chat transcripts out of the synced data. Errors are rendered as in-panel dashed “note” bubbles, never stack traces: a 429 shows a rate-limited note, a network/CORS fetch failure shows “Couldn’t reach Gemini…” with a prominent “Open full Gemini” link, a missing key shows an empty-state prompt and makes no request, and any other non-OK response shows the status plus error.message. Note bubbles use innerHTML only for the trusted link string; all user/model text is inserted via textContent. The header “Open full Gemini ↗” link (geminiLinkOut) opens gemini.google.com and works even with no key set. Supporting functions: loadChatHistory, saveChatHistory, openChatPanel, closeChatPanel, renderChatMessages, makeChatBubble, appendTransientBubble, setChatSending, clearChat, autoGrowChatInput. New state: state.geminiKey, state.geminiModel, state.chatHistory, state.chatSending. The panel and FAB are styled entirely from existing theme tokens, so light/dark and the per-persona accent apply automatically (including a body.light-mode backdrop override and a mobile full-width rule).

The Stack

Every language, framework, and service this project depends on, with the reasoning for each pick.

  • HTML / CSS / JSSingle file, vanilla, no build step. Matches Tom's stated preference for "single-file HTML when possible".
  • Google FontsInter (UI) + JetBrains Mono (code/timestamps). Loaded with display=swap. The only external CSS dependency.
  • GitHub REST APIBackend-as-a-database via private gists. Free, durable, no third-party service to outlive Tom's GitHub account.
  • GitHub PagesFree static hosting at tecgunther.github.io/startdashboard/. One URL works on any browser, any device.
  • Open-MeteoFree weather API AND geocoding (city search). No key required. Used by Weather widget for forecasts and by the Clock widget config to map city names to IANA timezones.
  • CoinGeckoFree public crypto prices and coin search. No key required. Used by the Crypto widget. Quote endpoint also returns 24h percent change (dollar change derived client-side). Unchanged in v1.9.
  • Finnhub (since v1.9)Stock quotes and symbol search. Free tier with API key, 60 calls/min, no credit card. Endpoints: finnhub.io/api/v1/quote (one symbol per call) and finnhub.io/api/v1/search. Used by Markets and Watchlist widgets. Replaced Twelve Data in v1.9 — Twelve Data's free tier was 8 calls/min, and Tom's ~12 symbols routinely exceeded that and hit HTTP 429 rate-limit errors. Finnhub's 60/min clears that. Signup: finnhub.io/register.
  • Google Gemini API (since v1.11)Powers the “Ask Gemini” chat panel. Free tier with an API key (no credit card), generated at aistudio.google.com/app/apikey. Endpoint: generativelanguage.googleapis.com/v1beta/models/<model>:generateContent (non-streaming), called directly from the browser — no backend. Default model gemini-2.5-flash (overridable in Settings). One external service; the key lives per-browser in localStorage. If the browser ever blocks the direct call, the feature degrades to a friendly message plus a link-out to gemini.google.com.
  • Google Favicon serviceBest-effort favicon images for bookmark rows (www.google.com/s2/favicons). Easy to patch out if Tom prefers privacy.

Key Decisions

Decisions made during the build, with the reasoning preserved so future-Tom understands why.

GitHub gist as the data backend, not a third-party service

StartMe-style services bring ads and lock-in. Supabase or jsonbin work but add a dependency on a free tier that may go away. GitHub gists are durable as long as Tom's GitHub account exists.

decided 2026-05-27 during plan revision

Personal Access Token per browser, stored in localStorage

Avoids OAuth complexity in a static page. Each browser gets its own token; losing a device means revoking one token. Token sent only to api.github.com.

decided 2026-05-27

Per-persona widgets, backgrounds, accent colors, and visibility

The whole point of personas is mental separation. Each persona is fully independent. v1.2 added persona.hiddenWidgets so a persona can opt out of any widget entirely (e.g. an "AI Education" persona with zero widgets, just categories).

decided 2026-05-27 (personas) and 2026-05-27 evening (per-persona widget visibility)

Per-device default persona + Twelve Data API key (localStorage, not gist)

Work Mac opens to Work; iPhone opens to Personal. Default persona is a per-device preference. The Twelve Data API key is also per-browser (parallel to the GitHub PAT) because rate limits are per-key and we shouldn't sync credentials through the gist.

decided 2026-05-27 (default persona) and 2026-05-27 evening (TD key)

Bookmarklet over a browser extension

One JavaScript snippet works in every browser, forever. A browser extension means three separate builds and ongoing maintenance.

decided 2026-05-27 during planning

Backgrounds: CSS-only presets + image URL (no upload)

Image upload would mean storing base64 in the gist, bloating the JSON on every save. URL-based images keep the gist small.

decided 2026-05-27

Last-write-wins on sync conflicts

Single-user app; full conflict resolution is overkill. Two-browser conflict surfaces a toast; most recent save wins. 60s background poll detects gist updates from other browsers.

decided 2026-05-27

Cross-window message via localStorage for bookmarklet-add visibility (v1.1)

When the bookmarklet popup adds a bookmark to a persona other than the one the main window is viewing, the addition is invisible until reload. v1.1 fixed this by writing sd_last_added to localStorage; the main window's storage event listener surfaces a banner with "Switch to [persona]". The Switch button (v1.3) reloads the gist before switching so the just-added bookmark is in memory.

decided 2026-05-27 evening

City search (Open-Meteo geocoding) instead of manual lat/lon and IANA strings (v1.1)

Original Weather config asked for "Name, lat, lon"; Clock asked for IANA timezone strings. Neither is usable by a normal person. v1.1 added a search-based widget config that hits Open-Meteo's free geocoding endpoint — type a city, pick a match, coordinates/timezone stored invisibly. Same endpoint serves both widgets.

decided 2026-05-27 evening

Twelve Data + ETF proxies for Markets (v1.2)

Twelve Data is the only free-tier US stock API that works reliably from browsers (Yahoo Finance blocks CORS). Markets widget defaults to SPY/QQQ/DIA (ETFs that track S&P/Nasdaq/Dow) because the actual indices need a paid tier. 60-second client cache on quotes prevents burning the 8-req/min free limit on re-renders.

decided 2026-05-27 late evening

Clock zones as {name, tz} objects instead of bare IANA strings (v1.3)

Original schema stored zones as ['America/New_York', ...]. Adding "Atlanta" silently failed if NYC was already there (same TZ). Migrated to [{name, tz}, ...] so two cities sharing a timezone render as separate rows. Backward compat: old string format auto-migrates on load.

decided 2026-05-28

Background CSS rule: don't set background: on #background (v1.3 hotfix)

Original CSS had #background { background: var(--bg) } which overrode every .bg-aurora / .bg-synthwave / etc. due to ID-vs-class specificity. The gradients never rendered — the page was always solid fallback color from v1.0 through v1.2. Removed the conflicting declaration so the class can paint. Body's background: var(--bg) serves as the pre-class fallback.

decided 2026-05-28 (hotfix after Tom reported "backgrounds don't change")

Bookmark descriptions: hover popover + multi-line modal edit (v1.5)

v1.4 made descriptions always-visible inline below the title. v1.5 hides them by default to keep rows clean; an accent dot signals presence, hover reveals the full text in a popover (0.5s show delay, instant hide). Edit opens a modal with an auto-growing textarea for long reminders / asterisks / line breaks.

decided 2026-05-28

Light/dark theme via tokenized CSS variables, not per-component overrides (v1.6)

Rather than write a light-mode rule for every component, the dark surface colors were extracted into four new tokens (--modal-bg, --input-bg, --popover-bg, --on-accent) and added to the existing palette tokens. The entire light skin is one body.light-mode { ... } block that re-defines those tokens. One change covers every component; adding future themes (e.g. high-contrast) becomes a single new override block. Theme choice persists in localStorage.sd_theme per browser, parallel to the GitHub PAT and the per-device default persona.

decided 2026-05-28 (v1.6)

Light palette pulled from the workspace hub-design-brief (v1.6)

Rather than invent a new light palette, the dashboard's light mode uses the same colors (--bg: #FAFAF7, --bg-elev-1: #FFFFFF, --text-primary: #1A1A1A, --border: #E8E6E0) that PROJECT-HUB, MASTER-HUB, TECHNICAL, and USER-EXPERIENCE already use. Visual consistency across Tom's whole workspace. Per-persona accent and accent-soft are not overridden — they're set per-persona at runtime, and preserving the persona color in both themes was an explicit plan decision.

decided 2026-05-28 (v1.6)

Light mode overrides the per-persona background with .bg-light-surface (v1.6)

The per-persona gradients (Aurora, Synthwave, Midnight Mesh, etc.) were designed against a dark background; in light mode they'd clash. The toggle removes the persona's .bg-* class from #background and adds .bg-light-surface instead; on toggle back to dark, applyPersonaTheme() restores the persona's gradient. There is deliberately no per-persona light-mode gradient option — making each persona configure a light variant of its background would roughly double the config burden for marginal benefit.

decided 2026-05-28 (v1.6)

formatPrice simplification: 2 decimals everywhere ≥ $0.01, 4 decimals below (v1.6)

v1.5 and earlier could render $92.232089 and -$2409.745305 on the finance widgets. Tom wanted prices that look like prices. New rule: n ≥ 0.01 → '$' + n.toLocaleString('en-US', { minimumFractionDigits: 2, maximumFractionDigits: 2 }); n < 0.01 → '$' + n.toFixed(4) so sub-cent altcoins don't display as $0.00. The previous special case for n ≥ 1000 is gone — toLocaleString handles thousand separators on its own. Applies uniformly to Crypto, Markets, and Watchlist since all three call formatPrice().

decided 2026-05-28 (v1.6)

Bookmark URL shown inside the popover, not in the browser's title tooltip (v1.6)

The bookmark <a> previously had title.title = b.url, which surfaces the browser's native URL tooltip on hover. That tooltip overlapped/blocked the description popover. Fix: remove the title attribute entirely; render the URL inside the popover on a small monospace footer line (.bk-desc-url) below the description text (.bk-desc-text). The separator border between description and URL uses an adjacent-sibling selector so it only appears when there's a description above — URL-only popovers render with no separator. Side-effect by design: the browser's native URL tooltip is gone from bookmarks; the URL lives in the popover.

decided 2026-05-28 (v1.6)

Negative-aware formatters branch on Math.abs() and re-apply the sign at output (v1.7)

v1.6's formatPrice() compared the raw value to 0.01. Any negative dollar change fell through to the sub-cent 4-decimal branch (-89.4463 → -$89.4463). v1.7 rewrites it: branch on Math.abs(n), format the magnitude with toLocaleString (or toFixed(4) for sub-cent), then prepend a '-$' or '$' sign string. Both call sites (renderCryptoWidget and renderTwelveDataWidget) lost their old .replace('$-', '-$') band-aids; each now reads as plain formatPrice(dollarChange). Reusable lesson: any future number formatter that accepts negative inputs should branch on Math.abs() and re-apply the sign — never compare the raw value to a positive threshold.

decided 2026-05-28 (v1.7 hotfix)

Active persona tab uses solid accent + --on-accent text, not --text-primary (v1.7)

The v1.6 active-tab style used var(--accent-soft) as the background, which was nearly invisible against most personas — on initial page load, no tab looked selected. v1.7 swaps to solid var(--accent) with font-weight: 600. The plan called for color: var(--text-primary), but against a solid accent fill that loses contrast in light mode. Used var(--on-accent) instead (the v1.6 token that's near-black in dark mode and white in light mode, specifically designed to ride on top of the accent color). The active-tab dot also moved to --on-accent so it stays visible against the now-solid tab. Separately: render() was calling renderPersonaTabs() before currentPersona(), but currentPersona() lazily initializes state.currentPersonaId on first call — so the first frame had no active tab regardless of styling. Fix is the ordering swap, with a comment in render() flagging the constraint. Note (v1.8): superseded by the soft-fill restyle below — the ordering fix and font-weight: 600 from v1.7 are preserved.

decided 2026-05-28 (v1.7 hotfix)

Active persona tab: soft-accent fill + accent outline + bold primary text (v1.8)

v1.7's solid var(--accent) fill with var(--on-accent) text solved the discoverability bug ("no tab selected on first frame") but overshot — the solid slab read as visually loud against the rest of the row. v1.8 keeps font-weight: 600 but softens the colors: background var(--accent-soft), text var(--text-primary), border-color var(--accent) (unchanged) for an accent outline. The active-tab dot reverted from var(--on-accent) back to var(--accent) so it reads correctly on the soft fill (opacity: 1 preserved). Same discoverability win, calmer look — the bold weight is doing as much of the lifting as the color now. Comment above the rule was rewritten to describe the new soft-fill + accent-outline intent.

decided 2026-05-28 (v1.8 hotfix)

Empty Settings input means “no change” for credentials (v1.8)

v1.7's saveSettings read the Twelve Data key input and, on an empty trimmed value, called localStorage.removeItem(LS.TWELVE_DATA_KEY) and set state.twelveDataKey = null. Result: every time Tom opened Settings to change something unrelated and clicked Save, the API key got nuked — because Safari's type="password" field can't reliably re-populate from autofill, so the field reads empty even when the key is stored. v1.8 deletes that destructive else branch entirely. Empty input = no change. Only a non-empty trimmed value updates the stored key. To explicitly clear the key, the user goes to Danger Zone → Reset browser, where destructive intent is unambiguous. A code comment above the block calls out the Safari rationale so this doesn't get reverted. Reusable principle: any future credential UI in this app (or anywhere else) should treat an empty save-time input as “no change,” not as “clear.” Make destructive actions explicit; saves should never be accidentally destructive.

decided 2026-05-28 (v1.8 hotfix)

Finance widgets retry once after 2 seconds on initial-load failure (v1.8)

Tom kept seeing red “Could not load” states on cold tab opens that resolved on a second manual reload — the classic symptom of a transient first-fetch flake (CoinGecko / Twelve Data hiccup, or a half-warm network on tab open). Rather than push the cache or block the render, v1.8 wraps the fetch + render body in renderCryptoWidget and renderTwelveDataWidget in an internal doFetch(isRetry) async helper. On the first failure (isRetry === false), the widget restores its “Loading prices…” / “Loading quotes…” placeholder and schedules setTimeout(() => doFetch(true), 2000). On the second failure (isRetry === true), it falls through to the existing fin-error message. Exactly one retry per widget per load, at 2000ms. Both Markets and Watchlist inherit this automatically since they both route through renderTwelveDataWidget. The existing _tdCache / TD_CACHE_MS logic in fetchTwelveDataQuotes is untouched — retry sits above the cache layer, so cached quotes still serve normally. Why one retry and not two: two would mask real outages too long; zero leaves manual-reload friction in place. One retry at 2 seconds is the sweet spot. Non-obvious side-effect: a sustained “Loading…” for ~2 seconds is now expected behavior on flaky cold opens (it's the retry running), not a hang — captured as a troubleshooting note.

decided 2026-05-28 (v1.8 hotfix)

Finnhub over Twelve Data for stock data (v1.9)

Twelve Data's free tier allows only 8 price lookups per minute. Tom tracks roughly 12 symbols across Markets and Watchlist, so a single dashboard load routinely exceeded the limit and returned HTTP 429 ("too many requests") — the chronic error he was seeing. Finnhub's free tier allows 60 calls per minute (also no credit card), which clears the headroom. The swap removed fetchTwelveDataQuotes in favor of fetchFinnhubQuotes(symbols) (one /quote call per symbol, normalized from Finnhub's c/d/dp fields) and moved symbol search to Finnhub's /search endpoint. The LS.TWELVE_DATA_KEY constant and state.twelveDataKey are kept for backward-compat reading only; nothing in app logic uses them anymore. Tom action: generate a free key at finnhub.io/register and paste it in Settings; the old Twelve Data key is ignored.

decided 2026-05-28 (v1.9)

Shared 5-minute localStorage quote cache, not per-tab in-memory (v1.9)

v1.8 and earlier cached quotes in a per-tab in-memory object (_tdCache), so every tab and every reload re-fetched independently — multiplying API calls and pushing toward the rate limit. v1.9 replaces it with sd_quote_cache in localStorage (5-minute TTL, shape { SYM: { price, dollar, percent, t } }, via readQuoteCache() / writeQuoteCache()). The cache is shared across all tabs and reloads; only stale or missing symbols hit the network. Opening the dashboard in three tabs no longer triggers three rounds of calls.

decided 2026-05-28 (v1.9)

On a rate limit (429), back off 60 seconds instead of retrying immediately (v1.9)

fetchFinnhubQuotes throws an error tagged isRateLimit when every symbol is rate-limited and nothing is cached. renderStockWidget shows "Rate limit reached — prices refresh shortly." and schedules a single refresh 60 seconds later. The v1.8 cold-start 2-second retry was dropped for stocks — retrying immediately under a rate limit just piles on more calls and makes the limiting worse. Per-symbol failures now degrade to "—" rather than triggering a retry. (Crypto / CoinGecko keeps its v1.8 single retry; it wasn't the source of the 429 pain.)

decided 2026-05-28 (v1.9)

Visible ⠿ grip handles because bare HTML5 drag was undiscoverable (v1.9)

Reordering already worked via native HTML5 drag, but there was no visual affordance — Tom couldn't tell what was draggable or where to grab. v1.9 prepends a visible ⠿ grip handle (.grip, grab cursor, brightens on hover) to categories (in renderCategory), bookmarks (in renderBookmark), and every config list item (via the shared makeCfgItemDraggable helper, applied to markets/watchlist tickers, weather cities, and clock zones). Drop position is shown with an accent line (.drop-before / .drop-after) and computed from whether the cursor is over the top or bottom half of the target. Cross-category and within-category bookmark moves are unified in moveBookmark(bkId, fromCatId, toCatId, beforeId, after); a drop on empty category space still appends.

decided 2026-05-28 (v1.9)

Multiple watchlists via an array + composite watchlist:<id> keys (v1.10)

The single widgets.watchlist object became persona.watchlists, an array of { id, title, symbols[] }. Each watchlist is its own card, addressed everywhere by a composite key 'watchlist:' + watchlist.id. That key is the single mechanism that lets each watchlist take part in widget order, hide/show, per-card color, the visibility toggles, and the config modal — exactly like a built-in widget — without scattering watchlist-specific branches through the codebase. Migration (ensurePersonaSchema + migrateWidgetOrder) converts the legacy singleton to one watchlist titled “Watchlist” (symbols preserved, old key deleted); new personas get one empty “Watchlist.” Any legacy bare 'watchlist' key found in widgetOrder/hiddenWidgets/widgetColors is remapped to the composite key(s). Verified against pre-widget-order, mid-version, and empty personas.

decided 2026-05-28 (v1.10)

Widget row converted from static HTML to fully dynamic rendering (v1.10)

Through v1.9 the widget cards were fixed markup in index.html. v1.10 needs both reorder and multiple instances of the same widget type (watchlists), neither of which a static layout supports. So renderWidgets now empties #widgets-row and rebuilds every card from persona.widgetOrder via buildWidgetCard. Order changes are a matter of rewriting that array (moveWidget). Trade-off: the Notes textarea is regenerated on every render — it keeps its notes-textarea id so existing references hold — and closing a widget config now triggers a full render() so edits propagate everywhere at once. This is the structural change that makes Features 1 and 2 possible.

decided 2026-05-28 (v1.10)

Markets kept as a singleton, not folded into watchlists (v1.10)

Markets tracks index proxies (SPY/QQQ/DIA for S&P/Nasdaq/Dow) — a fixed “how are the markets doing” view. A watchlist is a user-curated list of individual holdings. They serve different purposes, so Markets stays one un-renamable, non-deletable card while watchlists are an arbitrary, renamable, deletable set. Collapsing them into one concept would have muddied both.

decided 2026-05-28 (v1.10)

Per-card color stored in persona.widgetColors map + category.color, not scattered (v1.10)

Rather than hang a color field off each individual widget's own config object, all widget colors live in one per-persona map (widgetColors: { widgetKey → hex }), keyed by the same widget keys (including composite watchlist:<id> keys) used by order and visibility. Categories use category.color on the category object. An absent entry / null means “inherit the persona accent.” Keeping widget colors in one map means the color picker, the “Default (theme)” reset, and the migration logic all touch a single, predictable location. applyCardColor paints the border + soft glow; buildCardColorControl builds the shared picker (swatches + native color input + reset); openCategoryColorPopover reuses it for categories. The glow is a static box-shadow — deliberately no animation/pulse.

decided 2026-05-28 (v1.10)

Gemini Flash for the chat panel, not Claude or OpenAI (v1.11)

The “Ask Gemini” panel uses Google's Gemini Flash because it's free with no credit card, and it's the same model family behind the Google Search AI answers Tom already reaches for — so it fits the “fast quick answer” use case. Claude and OpenAI have no free tier, which would mean Tom paying per query for a convenience feature. The default model string is gemini-2.5-flash, but it's exposed as a Settings override field so that if Google renames or retires the model, Tom can switch without a code change.

decided 2026-05-28 (v1.11)

Browser-direct Gemini call with a graceful link-out fallback, not a backend (v1.11)

The dashboard has always been a zero-backend, single static file. Rather than stand up a proxy server to call Gemini, sendChatMessage calls generativelanguage.googleapis.com/…:generateContent directly from the browser with the key in an x-goog-api-key header. If the browser ever blocks that call (CORS), the panel shows a friendly “Couldn’t reach Gemini…” note with a prominent “Open full Gemini” link — the feature degrades gracefully instead of breaking. CORS caveat: the Build Agent could not definitively confirm the browser-direct path works without Tom's own key. The commonly-reported Gemini CORS error is about the OpenAI-compat endpoint (/openai/…), not the native :generateContent endpoint this build calls (which most browser tutorials hit directly). Tom's first real query is the empirical test; the fallback covers the worst case either way. A backend is deliberately not planned.

decided 2026-05-28 (v1.11)

Auto-retry Gemini’s transient 503 “overloaded” responses (v1.13)

Gemini occasionally returns an HTTP 503 “model overloaded” / “high demand” — a Google server-side capacity blip on the model tier, unrelated to the user’s quota, account, or time of day. These almost always clear within a second or two, so the standard remedy is a short automatic retry rather than surfacing the error. v1.11 treated a 503 as a dead end; v1.13 detects it (chatIsOverload(res): non-OK and either HTTP 503 or error text containing “overloaded”/“high demand”, case-insensitive) and retries up to CHAT_OVERLOAD_MAX_RETRIES = 2 times (1 initial + 2 = 3 attempts total) with backoffs CHAT_OVERLOAD_BACKOFFS = [1500, 3000] ms plus up to 500ms random jitter, keeping the “thinking…” indicator up throughout (no interim error flashes). If all retries fail, appendOverloadNote() shows a plain-English note (“Google’s servers are briefly overloaded — this is on Google’s side, not you. Try again in a moment.”) with a one-click Retry button and the existing link-out — never the raw API text. Limitation / deliberate scope: retries stay on the same model — there is no automatic model-switching fallback (the default gemini-2.5-flash tier is already relatively stable), and the cap is intentionally low (no 30s+ exponential backoff) so the chat stays responsive. A 503 is handled distinctly from a 429 rate limit (checked first, never retried — retrying a rate limit makes it worse) and from a network/CORS failure (link-out fallback, unchanged). To tune it, edit the CHAT_OVERLOAD_MAX_RETRIES / CHAT_OVERLOAD_BACKOFFS constants. The retry loop lives in runChatRequest (split out of sendChatMessage in v1.13 so the Retry button can re-issue the request without re-pushing the user’s message).

decided 2026-05-29 (v1.13)

Chat history per-device in localStorage, not synced to the gist (v1.11)

The conversation is stored in sd_chat_history (localStorage, capped to the last CHAT_HISTORY_CAP = 50 messages) so it survives reloads on that browser. It is deliberately not written to the synced gist: chat transcripts would bloat the synced data on every save, and which conversation you're mid-way through is naturally a per-device concern (the same reasoning behind the per-device theme and default-persona choices). “Clear chat” empties the state and removes the key.

decided 2026-05-28 (v1.11)

Quick-answer system prompt for concise replies (v1.11)

Gemini is primed with GEMINI_SYSTEM_PROMPT, a concise-answer instruction, so its default replies are short, direct, and search-style rather than long essays — matching the “fast answer without leaving the dashboard” intent. The request also sends only the last ~10 turns as contents, which keeps follow-up context without growing the payload unbounded.

How to Run

The launch command, the port number, and common troubleshooting.

To launch (production)

open https://tecgunther.github.io/startdashboard/

To preview locally (no sync)

Just double-click index.html, or:

open projects/personal-dashboard/index.html

The setup screen will appear since no PAT is in localStorage; you can paste a token to test sync locally if you want.

Troubleshooting

  • "Bad credentials" on first run — PAT is wrong or missing the gist scope. Regenerate at github.com/settings/tokens.
  • Sync indicator turns red ⚠ — click it to reload. If still failing, the token may have expired; reset this browser in Settings.
  • Cross-browser changes not showing — reload the other browser (Cmd+R). Background poll is 60s, but reload is instant.
  • Bookmarklet popup blocked — allow popups for tecgunther.github.io in browser settings.
  • Weather "Could not load" — transient Open-Meteo issue, or invalid lat/lon. Confirm coordinates in Settings.
  • Markets / Watchlist say "Add your Finnhub API key" (v1.9) — stock widgets use Finnhub now, not Twelve Data. Generate a free key at finnhub.io/register and paste it into Settings ⚙. The old Twelve Data key is ignored.
  • "Rate limit reached — prices refresh shortly" (v1.9) — every tracked symbol got rate-limited with nothing cached. The widget backs off and auto-refreshes once after 60 seconds. No action needed; Finnhub's free tier is 60 calls/min.
  • A single stock row shows "—" (v1.9) — that one symbol's /quote call failed (bad ticker or a transient error). The rest of the widget still renders; the dash means "no data for this symbol," not a widget-wide failure.
  • Crypto widget hangs on "Loading…" for ~2 seconds (v1.8) — that's CoinGecko's cold-start retry doing its job. The first fetch failed transiently and the widget auto-retries once at 2000ms before falling through to a red error. (Stocks no longer do this immediate retry as of v1.9.)
  • Need to explicitly clear the Twelve Data API key (v1.8) — the Settings field can no longer remove it (empty means “no change”). Use Settings → Danger Zone → Reset browser.

How to Modify

"If you want to change X, look in Y." A quick map for future edits.

  • Add a new background preset. Add a CSS class .bg-<name> in the <style> block (search "Background presets") and a matching entry to BG_PRESETS in the script. The picker grid and applyPersonaTheme render dynamically from BG_PRESETS / swap the .bg-<id> class, so no other wiring is needed. There are 10 presets as of v1.14 (the newest is starlink, a CSS-only deep-space scene). A preset may carry an optional accent field (added in v1.14, e.g. Starlink’s accent:'#4F9CFF'): when a preset with an accent is clicked in the background-thumbnail grid, the persona accent input is set to that color (live preview) and Save persists it to persona.accent — overridable afterward. Presets without an accent leave the accent untouched. To make a preset suggest an accent, just add the accent field; to make one not, omit it.
  • Add a new widget. Add a <div class="widget" data-widget="<name>"> in the widgets-row section, append the name to ALL_WIDGETS and WIDGET_LABELS, write a render<Name>Widget(persona) function, call it from renderWidgets(), and (if it needs config) extend openWidgetConfig, renderWidgetCfgList, and dispatchWidgetSearch.
  • Change the dark color palette. CSS custom properties at the top of the :root block. Tokens include the v1.6 additions --modal-bg, --input-bg, --popover-bg, --on-accent.
  • Change the light palette. The body.light-mode { ... } block near the top of the <style> section — re-defines the same tokens with light values. Touch nothing else.
  • Add a new theme (e.g. high-contrast). Add a new body class (body.high-contrast-mode) with its own token overrides, add a state value to state.theme, extend applyTheme() to swap the right class on <body>, and add the corresponding cycle in toggleTheme(). No per-component CSS changes needed if you stay inside the tokens.
  • Restore the browser URL tooltip on bookmarks. Re-add title.title = b.url inside renderBookmark(). (Note: this re-introduces the v1.5 overlap with the description popover.)
  • Change the gist filename. Constant GIST_FILENAME. Don't change after first deploy — silently breaks sync.
  • Disable the favicon service. Search s2/favicons and remove the block or swap for another source.
  • Change the search engine list. The SEARCH_ENGINES object near the top of the script.
  • Change the bookmarklet popup size. Generated in openBookmarkletModal(); width=520,height=600 values are inline.
  • Change the stock quote cache TTL (v1.9). QUOTE_CACHE_MS (default 5 minutes). It governs the shared sd_quote_cache localStorage cache read/written by readQuoteCache() / writeQuoteCache(). Lower = fresher prices, but more Finnhub calls (and closer to the 60/min limit).
  • Adjust cross-add banner timeout. The 5-minute window in checkRecentAddition().
  • Adjust hover-popover show delay. The transition-delay: 0.5s in .bookmark:hover .bk-desc-popover.
  • Style the popover URL footer. The .bk-desc-url class (URL footer) and .bk-desc-text (description body) live in the popover CSS block. The separator border between them is on .bk-desc-text + .bk-desc-url (adjacent sibling) so URL-only popovers have no separator.
  • Adjust the price decimals rule. formatPrice(n) — branches on Math.abs(n) (the >= 0.01 branch uses toLocaleString; the sub-cent branch uses toFixed(4)) and re-applies the sign via a '-$' / '$' prefix at output. Don't compare the raw n to a positive threshold — negatives will fall through to the wrong branch.
  • Tweak the active persona tab style. .persona-tab.active (v1.8) — background is var(--accent-soft), text is var(--text-primary), border-color is var(--accent) (the accent outline), and the dot inside it is var(--accent). font-weight: 600 carries a lot of the visual weight — keep it. If you ever go back to a solid-accent fill (the v1.7 look), switch text to var(--on-accent) and the dot to var(--on-accent) at the same time so contrast holds in both themes.
  • Adjust the bookmark description popover spacing. .bookmark .bk-desc-popover — margin-top: 28px. The gap is intentional: Safari and some Chromium builds render their own tooltip for ellipsis-truncated titles, and there's no reliable cross-browser way to suppress it. Don't shrink the gap unless you've verified across all of Tom's browsers.
  • Change the stock data source (v1.9: Finnhub). The key in Settings is sd_finnhub_key (set-finnhub-key field). Known Limitations

What the project doesn't do, what it can't do without significant work, what's brittle.

  • Real-time collaboration is not supported. Sync polls every 60 seconds and saves are debounced ~1.5 seconds. If two browsers edit at once, last save wins and a toast surfaces the conflict.
  • Image uploads are not supported in v1. Backgrounds can be presets or URL-based images. Adding upload would mean base64-encoding into the gist, which bloats it on every save.
  • No automatic bookmark import. Bookmarks come over manually as Tom uses the bookmarklet. Adding a JSON-import flow is straightforward if needed later.
  • PAT lives in localStorage. Anyone with physical access to an unlocked browser could read it. For a personal-use app on a personal machine this is acceptable. For shared devices, log out / lock the screen.
  • Gist is private but not encrypted. GitHub staff and GitHub's automated systems could, in principle, see the contents. The data is bookmarks and notes — not secrets — so this is acceptable.
  • The favicon service sends bookmark domains to Google. If this matters to you, see "How to Modify" above.
  • Bookmarklet popups can be blocked. Most browsers allow them for the originating site by default, but some configurations block them entirely. Tom can allow popups for tecgunther.github.io if it happens.
  • No undo. Delete a bookmark and it's gone. Worth adding a "Recently deleted" trash bin in a future iteration.
  • No per-persona light backgrounds (v1.6). Light mode replaces the persona's gradient with one shared .bg-light-surface. Personas keep their accent color in light mode, but they lose their unique background. Making each persona configure a light variant of its background was deliberately deferred — would roughly double the config burden for marginal benefit. If a future Tom wants per-persona light backgrounds, the hook is in applyTheme() + applyPersonaTheme().
  • No native browser URL tooltip on bookmarks (v1.6). By design — the title attribute was removed so it doesn't overlap the description popover. The URL is now visible on a small footer line inside the popover instead.
  • The browser's native tooltip for ellipsis-truncated bookmark titles can't be suppressed cross-browser (v1.7). Safari (and some Chromium builds) show their own tooltip for titles that have been visually truncated with text-overflow: ellipsis, even when there's no title attribute. There's no reliable way to suppress it across browsers, so the description popover is positioned margin-top: 28px below the bookmark row to leave space for it rather than fight the browser. Side-effect: a touch of extra vertical whitespace under each bookmark row when the popover is open.
  • Finnhub free tier: one symbol per call, US-focused (v1.9). Finnhub's free /quote endpoint returns a single symbol per request, so Markets + Watchlist fan out one call per tracked ticker. The 5-minute shared cache keeps this comfortably under the 60-calls/min free limit for Tom's ~12 symbols, but a much longer watchlist could approach it. Coverage is strongest for US equities and major ETFs; some international tickers may not resolve on the free tier.
  • Crypto has no shared quote cache, and re-fetches on every full render (v1.9 / more frequent in v1.10). The 5-minute shared localStorage cache (sd_quote_cache) added in v1.9 covers stock quotes only. The Crypto widget (CoinGecko) still fetches on every render and keeps its v1.8 single retry. This was a non-issue when full re-renders were rare, but v1.10's drag-to-reorder triggers a full render() on each drop, so the Crypto fetch now fires more often. It's pre-existing behavior, not a v1.10 regression, and CoinGecko's free tier hasn't yet been a rate-limit problem — but if rate-limiting ever appears, the fix is to add a shared CoinGecko cache mirroring the 5-minute Finnhub one (readQuoteCache/writeQuoteCache pattern). Tracked as an open item.
  • API key (now Finnhub) cannot be cleared from the Settings field (v1.8 behavior, carried into v1.9). An empty trimmed value in the Settings input means “no change,” not “clear.” This is intentional — Safari's type="password" fields can't reliably re-populate from autofill, so before v1.8 every Settings save with an empty key field silently wiped the stored key. To explicitly clear the key, use Danger Zone → Reset browser, which is unambiguous about destructive intent. Trade-off: there's no "just clear this one credential" path short of Reset browser, which wipes the whole local setup for that browser. Acceptable cost for the safety win.
  • The Ask Gemini chat is a fresh assistant, not Tom's personal Gemini account (v1.11). It has no memory of Tom's real Gemini conversations, no access to his Google account context or logged-in features — it's an anonymous API call primed by a system prompt. For his actual account, the “Open full Gemini” link-out is the path.
  • The chat is non-streaming (v1.11). The reply appears all at once when Gemini finishes, rather than token-by-token. Streaming was deferred to avoid the streaming endpoint's separate CORS wrinkle; for short quick answers the difference is small.
  • Browser-direct Gemini works — CORS confirmed (v1.11, verified v1.13). The chat calls Gemini directly from the browser. Tom confirmed the native :generateContent endpoint is not CORS-blocked (the commonly-reported Gemini CORS error is about the OpenAI-compat endpoint, which this build does not use), so the v1.11 open question is closed. The graceful “Couldn’t reach Gemini” message plus link-out remains as the fallback for any future connection failure — a backend is deliberately not planned. Separately, as of v1.13 a transient Google-side 503 “model overloaded” auto-retries (up to 2x, short backoff) rather than surfacing an error; see the decision above.
  • Chat history is per-device only (v1.11). The conversation lives in this browser's localStorage and is not synced through the gist, so opening the dashboard on another device starts a fresh chat. This is by design (transcripts would bloat the synced data), but it means there's no single, cross-device chat history.
