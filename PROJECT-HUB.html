How to Run This App

Key URLs

  • Live dashboard: https://tecgunther.github.io/startdashboard/
  • GitHub repo (for code edits): github.com/TECGUNTHER/startdashboard
  • GitHub Pages settings: github.com/TECGUNTHER/startdashboard/settings/pages
  • Personal Access Tokens (to revoke or regenerate): github.com/settings/tokens
  • Finnhub signup (stock widgets): finnhub.io/register
  • Gemini API key (the Ask Gemini chat, v1.11 — free, no card): aistudio.google.com/app/apikey → paste into Settings → Gemini API key
  • Find your gist ID: open the dashboard → Settings (⚙) → Sync → Gist ID

One-time setup (already done)

The project is deployed and the first browser is set up. To repeat this whole sequence on a fresh machine, see README.md.

git init && git add . && git commit -m "Initial StartDashboard"
git remote add origin https://github.com/TECGUNTHER/startdashboard.git && git push -u origin main

Then in GitHub: Settings → Pages → Deploy from a branch → main / (root).

To launch

Open in any browser:

open https://tecgunther.github.io/startdashboard/

For local-only preview (no sync), just double-click index.html.

No port required — static page served by GitHub Pages.

Troubleshooting

  • "Bad credentials" on first run · PAT was pasted wrong or is missing the gist scope. Regenerate.
  • Sync indicator turns red · Click it to reload from the gist. If still failing, the token may have expired — regenerate and reset this browser in Settings.
  • Bookmarklet popup blocked · Allow popups for tecgunther.github.io in browser settings.
  • Changes not appearing across browsers · Reload the other browser. Sync polls every ~60s; reload is instant.
  • Markets / Watchlist say "Add your Finnhub API key" · As of v1.9, stock widgets use Finnhub, not Twelve Data. Generate a free key at finnhub.io/register and paste it into Settings. The old Twelve Data key is ignored.
  • "Rate limit reached" on Markets / Watchlist · Finnhub’s free tier is 60 calls/min. The widget backs off and auto-refreshes once after 60 seconds — no action needed.
  • Ask Gemini chat says "Add your free Gemini API key" · No key set. Generate one free at aistudio.google.com/app/apikey and paste it into Settings → Gemini API key. The link-out button still works without a key.
  • Chat says "Couldn’t reach Gemini" · Either a network hiccup, a rate limit, or the browser blocked the direct call (CORS). Use the "Open full Gemini" link in the note to continue in your real Gemini account.

v1.11 — a built-in “Ask Gemini” AI chat panel inside the dashboard: floating bubble → slide-over panel, concise quick answers, per-device history, and a link-out to the full Gemini. Real Build Agent + Documentation Agent subagent flow.

Completed

  • In-dashboard AI chat panel. A floating chat bubble (bottom-right, theme- and accent-aware) opens a slide-over “Ask Gemini” panel from any persona. Header has the title, the model label, an “Open full Gemini ↗” button, and a close ✕; below that a scrollable message history, an input row (Enter sends, Shift+Enter for a newline, Escape/backdrop/✕ closes), and a “Clear chat” footer. The bubble is hidden on the setup and add-bookmark screens and shown only on the main dashboard. New functions include openChatPanel, closeChatPanel, renderChatMessages, sendChatMessage, and clearChat.
  • Powered by Google’s free Gemini API, browser-direct, no backend. Sends a POST to generativelanguage.googleapis.com/…:generateContent with the key in a header; default model is gemini-2.5-flash. It passes a system instruction that primes short, direct, search-style answers, plus the last ~10 turns for follow-up context. This keeps the dashboard’s zero-backend design intact.
  • Quick-answer tuning. A concise-answer system prompt (GEMINI_SYSTEM_PROMPT) makes replies short and direct by default — the same fast-answer feel as Google Search’s AI, rather than long essays.
  • Per-device chat history. The conversation is stored in this browser’s localStorage (capped at the last 50 messages) so it survives reloads, but it is deliberately not synced through the gist — chat transcripts would bloat the synced data, and history is genuinely a per-device thing. “Clear chat” empties it.
  • “Open full Gemini” link-out. The header button opens gemini.google.com for Tom’s real, logged-in Gemini account — works even with no key set. The in-panel chat is a separate, fresh assistant with no access to that account’s history.
  • Settings: Gemini key + optional model field. A new “Gemini API key” password field (with an inline AI Studio walkthrough link — free, no card) and an optional model-override field sit just after the Finnhub key. Both follow the v1.8 non-destructive save rule: an empty field means “no change,” and an empty model falls back to the default.
  • Friendly error handling. A 429 shows a rate-limited note; a network/CORS failure shows “Couldn’t reach Gemini…” with a prominent “Open full Gemini” link; a missing key shows an empty-state prompt and makes no request. All errors render as in-panel notes — never a stack trace. All chat text is inserted as plain text; only the trusted link string uses markup.
  • No new dependencies. Only index.html changed (grew to ~5,190 lines). New external service: Google Gemini (free tier). Markets/Finnhub, Crypto/CoinGecko, Weather/Open-Meteo, and gist sync are all unchanged.

Decisions made

  • Gemini Flash, not Claude or OpenAI. It’s free with no credit card, and it’s the same model family behind the Google Search AI answers Tom already reaches for. Claude and OpenAI have no free tier, and the use case here is a fast quick answer — which Flash fits best.
  • Browser-direct call with a graceful link-out fallback, instead of adding a backend. Keeps the dashboard’s zero-backend architecture. Worst case (if the browser blocks the call) is a friendly message plus the “Open full Gemini” link — not a broken feature.
  • Chat history per-device, not synced to the gist. Transcripts would bloat the synced data, and which conversation you’re mid-way through is naturally a per-device thing.
  • Concise system prompt. Tunes the assistant toward short, search-style answers by default rather than long-form responses.
  • Model is a Settings override. The model name lives in a Settings field (defaulting to gemini-2.5-flash), so if Google renames or retires the model Tom can switch without a code change.
  • CORS confirmed only on first real use. The Build Agent couldn’t definitively prove the browser-direct :generateContent call works without Tom’s own key. The commonly-reported Gemini CORS error is about the OpenAI-compat endpoint, not the native endpoint this build calls — but Tom’s first query is the real test. The fallback makes a worst case graceful. (Logged as an Open Item.)

Completed

New helpers addWatchlist, deleteWatchlist, renderStockWidgetFor.

New helpers applyCardColor, buildCardColorControl, openCategoryColorPopover; new CSS classes .card-color-control, .card-color-swatches, .card-color-row, .color-popover.

Any legacy bare 'watchlist' key is remapped to the composite key(s); missing built-in widgets and new watchlists are appended to the order; stale watchlist keys are dropped. Decisions made

  • Watchlist rename updates live as you type — immediate feedback, no separate save step for the title.
  • Closing a widget config now does a full re-render so any title/color/symbol change is reflected everywhere at once.
2026-05-28 · Thursday · later evening
v1.9 — migrated stock data from Twelve Data to Finnhub (fixes the chronic rate-limit errors), added a shared 5-minute price cache, and overhauled drag-to-reorder with visible grip handles.

Completed

  • Finnhub replaces Twelve Data for all stock data. Why: Twelve Data’s free tier caps at 8 price lookups per minute, and Tom’s ~12 tracked symbols routinely blew past that — producing the constant HTTP 429 (“too many requests”) errors he was seeing. Finnhub’s free tier allows 60 calls a minute (no credit card). The old fetchTwelveDataQuotes was removed and replaced with fetchFinnhubQuotes(symbols), which makes one /quote call per symbol (fanned out in parallel) and normalizes Finnhub’s c/d/dp fields into the dashboard’s existing price/dollar/percent shape. If one symbol fails it shows “—” rather than killing the whole widget. Symbol search moved to Finnhub’s /search endpoint; the ETF proxies (SPY/QQQ/DIA) still appear. The Settings field, its inline walkthrough, and all empty-state messages now point at Finnhub (finnhub.io/register). The old Twelve Data key is still read for backward-compat but is no longer used by anything.
  • Shared 5-minute price cache. Replaced the old per-tab in-memory quote cache with a localStorage-backed cache (sd_quote_cache, 5-minute TTL) that every tab and reload draws from. Only stale or missing symbols hit the network — opening the dashboard in three tabs no longer triggers three rounds of API calls. Helpers readQuoteCache() / writeQuoteCache() manage it.
  • Smarter rate-limit handling. If every symbol is rate-limited with nothing cached, the widget shows “Rate limit reached — prices refresh shortly” and schedules a single refresh 60 seconds later. The v1.8 immediate 2-second retry was dropped for stocks — it actually made rate-limiting worse. (Crypto / CoinGecko keeps its v1.8 single retry and was otherwise left alone.)
  • Drag-to-reorder overhaul with visible grip handles. The old HTML5 drag was undiscoverable — nothing showed Tom what to grab. Now categories, bookmarks, and every config list (stock tickers, weather cities, clock zones) prepend a draggable ⠿ grip handle (a grab cursor, brightens on hover). A blue drop-indicator line shows where the item will land. Bookmark drops are position-aware (drop on the top or bottom half of a row to insert before/after) via a new moveBookmark() helper that handles both within-category and cross-category moves; dropping on empty category space still appends. Config-list reordering is handled by a new shared makeCfgItemDraggable() helper.

Decisions made

  • Finnhub over Twelve Data — 60 calls/min vs 8 on the free tier. Tom’s symbol count made Twelve Data’s limit a constant source of 429s; Finnhub clears that headroom. Both are free, so no cost change.
  • Shared localStorage quote cache, not per-tab in-memory — multiple tabs and reloads were multiplying API calls. A 5-minute cache shared across tabs cuts that down and keeps Tom comfortably under the rate limit.
  • On 429, back off 60 seconds instead of retrying immediately — v1.8’s 2-second retry made rate-limiting worse by piling on more calls. One delayed refresh is gentler and recovers cleanly.
  • Visible ⠿ grip handles — bare HTML5 drag had no affordance; Tom literally couldn’t find anything to grab. An explicit grip makes the drag target obvious.
  • Per-symbol graceful degradation — dropped the stock widget’s cold-start retry; a single failed symbol now shows “—” instead of taking down the widget. Skipped adding a localStorage cache for crypto to keep this change focused.

2026-05-28 · Thursday · evening
v1.8 hotfix — API key persistence, cold-start retry on finance widgets, and a calmer active-tab style (three small fixes on top of v1.7), plus a new install guide for third-party users.

Completed

  • Settings no longer wipes the Twelve Data API key when the field is left empty. In v1.7, saveSettings read the input, and on an empty trimmed value it called localStorage.removeItem(LS.TWELVE_DATA_KEY) and set state.twelveDataKey = null — meaning every time Tom opened Settings to change something else and hit Save, the API key got nuked (Safari's type="password" fields don't reliably re-populate from autofill). v1.8 deletes that destructive else branch entirely: empty input means “no change.” Only a non-empty trimmed value updates the stored key. A comment above the block calls out the Safari rationale and points users at Danger Zone → Reset browser if they ever genuinely need to clear the key.
  • Cold-start retry on Crypto, Markets, and Watchlist. Refactored renderCryptoWidget and renderTwelveDataWidget to wrap the fetch + render body in an internal doFetch(isRetry) async helper. On the first failure, the widget restores its “Loading prices…” / “Loading quotes…” placeholder and schedules setTimeout(() => doFetch(true), 2000). On the second failure (isRetry === true), it falls through to the existing fin-error message. Exactly one retry per widget per load, at 2000ms. Both Markets and Watchlist inherit this automatically since they both route through renderTwelveDataWidget. The _tdCache / TD_CACHE_MS logic in fetchTwelveDataQuotes is untouched — retry sits above that layer. Why this matters: Tom kept seeing red “Could not load” states on first load that resolved on a second manual reload — classic transient first-fetch flake on a cold tab.
  • Active persona tab restyled to a soft fill, not a solid slab. v1.7's solid var(--accent) fill with var(--on-accent) text was high-contrast but visually loud against the rest of the row. v1.8 keeps the font-weight: 600 bolding but dials the colors back: background is now var(--accent-soft), text is var(--text-primary), border keeps var(--accent) for the outline. The active-tab dot reverted from var(--on-accent) back to var(--accent) (it now reads on the soft fill instead of the solid slab); opacity: 1 on the dot is preserved. The comment above the rule was rewritten to describe the new soft-fill + accent-outline intent.
  • Install guide for third-party users added. New file INSTALL-FOR-OTHERS.md at the project root — a self-contained walkthrough someone else can follow to fork the repo, enable GitHub Pages on their own fork, generate their own PAT, and run their own private StartDashboard with their own gist. README stays Tom-specific (with a note at the top pointing newcomers at the new file). Fourth doc card added to PROJECT-HUB's Documentation row. Doc audit also ran clean: fixed three small staleness issues — the v1.0 weather-widget workflow in UX (replaced with the v1.1+ city-search flow), the “four widgets baked in” line (it's been seven since v1.2), and the TECHNICAL line-count claim (3,750 → 3,860).

Decisions made

  • Empty Settings input means “no change” for credentials. Don't make a save operation accidentally destructive. The only way to explicitly clear the API key is Danger Zone → Reset browser, where destructive intent is unambiguous. Reusable principle for any future credential UI.
  • Cold-start retry: one retry, 2000ms, then fall through to the existing error. Two retries would mask real outages too long; zero retries leaves Tom doing manual reloads for what's clearly a transient flake. One retry at 2 seconds is the sweet spot.
  • Soft-fill active tab over solid slab. The v1.7 solid fill solved discoverability (Tom couldn't tell which tab was active on cold load) but overshot — the slab read as visually loud. Soft fill + bold text + accent outline keeps the discoverability win without the loudness. font-weight: 600 is doing as much of the lifting as the color now.

New troubleshooting tips

  • Widget shows “Loading…” for ~2 seconds before resolving. That's the new cold-start retry doing its job — the first fetch failed transiently and the widget is automatically retrying once before falling through to an error. Not a hang.
  • Need to explicitly clear the Twelve Data API key? Settings → Danger Zone → Reset browser. The Settings field can no longer remove the key (empty means “no change”), specifically because Safari's password-type field can't reliably re-populate from autofill.

v1.7 hotfix — three small bug fixes on top of v1.6 (negative-price decimals, popover/tooltip overlap, active-tab highlight).

Completed

  • Negative finance prices now show 2 decimals. formatPrice() in v1.6 used a raw n >= 0.01 comparison, so any negative value fell through to the 4-decimal branch (-89.4463 → -$89.4463). Rewritten to branch on Math.abs(n) and re-apply the sign at output. Old .replace('$-', '-$') band-aids at the two call sites (renderCryptoWidget and renderTwelveDataWidget) are gone — each call site is just formatPrice(dollarChange) now. Verified against Tom's screenshot: -89.4463 → -$89.45, -2477.0102 → -$2,477.01, -0.13 → -$0.13.
  • Bookmark description popover no longer collides with the browser's native truncation tooltip. Safari (and some Chromium builds) show a native tooltip for ellipsis-truncated bookmark titles, even though the title attribute is gone. That tooltip was overlapping the description popover. CSS rule .bookmark .bk-desc-popover margin-top bumped from 4px to 28px, with a code comment noting the cross-browser reason — we leave room for the native tooltip rather than trying to suppress it.
  • Active persona tab is now highlighted on initial page load. Two fixes:
    • Ordering bug: render() called renderPersonaTabs() before currentPersona(), but currentPersona() lazily initializes state.currentPersonaId on first call. So on the very first frame, no tab had the .active class. Moved const p = currentPersona(); above renderPersonaTabs(); a comment now flags the ordering constraint.
    • Visual bug: the previous active style used var(--accent-soft) background, which read as nearly invisible against most personas. Now uses solid var(--accent) with font-weight: 600, and text color var(--on-accent) for contrast (the v1.6 token: near-black in dark mode, white in light mode). Active-tab dot also swapped to --on-accent so it stays visible against the now-solid accent fill.

Decisions made

  • Use --on-accent for active-tab text instead of --text-primary. The plan called for --text-primary, but against a solid accent fill that loses contrast in light mode. --on-accent is the token specifically designed to ride on top of the accent color in both themes.
  • Accommodate the browser's native truncation tooltip, don't try to suppress it. Safari ignores most suppression tricks for ellipsis tooltips. Pushing the popover down 28px is cheaper than trying to fight the browser.
  • Math.abs + sign re-application is the right pattern for any negative-aware formatter. Captured as a troubleshooting note in TECHNICAL so future formatters don't repeat the mistake.
