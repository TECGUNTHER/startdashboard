Recently completed

  • v1.16 shipped — rebuilt how you rearrange widget and category cards. Until now, dragging cards used the browser’s built-in drag, which was flaky — and after the v1.15 masonry change it sometimes guessed the wrong direction (it still thought cards were in a left-to-right row). v1.16 throws that out and replaces it with a smooth, custom drag: grab a card by its ⠿ grip and a faded copy follows your cursor, while a highlighted “landing slot” (a glowing dashed outline) shows exactly where the card will drop. Let go and it lands there reliably, every time. As a bonus, the same custom drag works on touch — so you can now rearrange cards on your iPhone, which the old browser drag never supported. Also new: hover-revealed ▲/▼ arrows on every widget and category card — if you’d rather not drag, just hover a card and click up/down to nudge it one spot. The native drag for bookmarks, persona tabs, and the config lists (tickers, cities, zones) is intentionally left as-is and still works exactly as before. Only index.html changed; no new dependencies.
  • v1.15 shipped — masonry (vertical-packing) layout for the widgets and the categories. Before this, the widgets sat in an equal-height grid: a tall card (say a long Watchlist) stretched its neighbors to match and shoved the categories far down the page. The same happened in the categories grid. Now both the widget row and the categories pack like Pinterest tiles — each card is only as tall as its own content, a tall card extends down while shorter cards pack up beside it, and the wasted space is gone (categories sit higher on the page). The drag-to-reorder grip handles (⠿) still work for both widgets and categories, and every per-card feature (glow colors, gear configs, multiple watchlists, all 10 backgrounds incl. Starlink, light/dark) is untouched. Pure CSS change — only index.html changed, no JS logic touched, no new dependencies.
  • v1.14 shipped — a new “Starlink” background. The background picker in Settings now has a 10th option, Starlink: a calm, cinematic deep-space scene — a near-black sky with a scattering of faint stars, a soft diagonal Milky Way band, and a few low-glow blue/teal/purple nebula washes. It’s pure CSS (no images, no animation), so it loads instantly. Picking it also auto-switches that persona’s accent to a matched Starlink electric-blue (#4F9CFF) — you see the change live in the picker, and it saves when you Save. You can still override the accent to anything you like afterward; picking any of the other 9 backgrounds leaves your accent alone, exactly as before. Per-persona and synced like every other background. Only index.html changed; no new keys or dependencies.
  • v1.13 shipped — the Ask Gemini chat now rides out Google’s overload blips. When Google’s servers briefly say “this model is experiencing high demand” (a transient 503 — Google’s side, not yours), the chat no longer shows you that error. It quietly retries up to twice (about 1.5s then 3s apart), keeping the “thinking…” indicator up, and most blips clear on the first retry. If it still can’t get through, you get a plain-English note — “Google’s servers are briefly overloaded — this is on Google’s side, not you. Try again in a moment.” — with a one-click Retry button and the “Open full Gemini” link. Rate-limit (429) and connection errors behave exactly as before.   • v1.11 shipped — a built-in “Ask Gemini” AI chat panel. A floating chat bubble in the bottom-right corner opens a slide-over panel where you can ask a question and get a concise, search-style answer without leaving the dashboard. Available from every persona. Powered by Google’s free Gemini API.
  • Tuned for quick answers. The assistant is primed (via a system instruction) to give short, direct, search-style replies rather than long essays — the same kind of fast answer you get from Google Search’s AI. Follow-up questions keep recent context (it sends the last ~10 turns).

How to Run This App

Key URLs

  • Live dashboard: https://tecgunther.github.io/startdashboard/
  • GitHub repo (for code edits): github.com/TECGUNTHER/startdashboard
  • GitHub Pages settings: github.com/TECGUNTHER/startdashboard/settings/pages
  • Personal Access Tokens (to revoke or regenerate): github.com/settings/tokens
  • Finnhub signup (stock widgets): finnhub.io/register
  • Gemini API key (the Ask Gemini chat, v1.11 — free, no card): aistudio.google.com/app/apikey → paste into Settings → Gemini API key
  • Find your gist ID: open the dashboard → Settings (⚙) → Sync → Gist ID

One-time setup (already done)

The project is deployed and the first browser is set up. To repeat this whole sequence on a fresh machine, see README.md.

git init && git add . && git commit -m "Initial StartDashboard"
git remote add origin https://github.com/TECGUNTHER/startdashboard.git && git push -u origin main

Then in GitHub: Settings → Pages → Deploy from a branch → main / (root).

To launch

Open in any browser:

open https://tecgunther.github.io/startdashboard/

For local-only preview (no sync), just double-click index.html.

No port required — static page served by GitHub Pages.

Troubleshooting

  • "Bad credentials" on first run · PAT was pasted wrong or is missing the gist scope. Regenerate.
  • Sync indicator turns red · Click it to reload from the gist. If still failing, the token may have expired — regenerate and reset this browser in Settings.
  • Bookmarklet popup blocked · Allow popups for tecgunther.github.io in browser settings.
  • Changes not appearing across browsers · Reload the other browser. Sync polls every ~60s; reload is instant.
  • Markets / Watchlist say "Add your Finnhub API key" · As of v1.9, stock widgets use Finnhub, not Twelve Data. Generate a free key at finnhub.io/register and paste it into Settings. The old Twelve Data key is ignored.
  • "Rate limit reached" on Markets / Watchlist · Finnhub’s free tier is 60 calls/min. The widget backs off and auto-refreshes once after 60 seconds — no action needed.
  • Ask Gemini chat says "Add your free Gemini API key" · No key set. Generate one free at aistudio.google.com/app/apikey and paste it into Settings → Gemini API key. The link-out button still works without a key.
  • Chat says "Couldn’t reach Gemini" · Either a network hiccup, a rate limit, or the browser blocked the direct call (CORS). Use the "Open full Gemini" link in the note to continue in your real Gemini account.

v1.16 — reliable card rearranging: replaced the flaky native drag for widget + category cards with a custom vanilla pointer-drag engine that shows a highlighted “landing slot,” and added hover-revealed ▲/▼ move arrows on every widget and category card. Fixes the v1.15 masonry wrong-axis bug and works on touch (iPhone). CSS + JS, only index.html. Real Build Agent + Documentation Agent subagent flow.

Completed

  • One shared pointer-drag engine for widget + category cards. A new initCardDrag({ grip, card, container, cardSelector, onDrop }) replaces the old native HTML5 drag on both card types. On pointerdown on a card’s ⠿ grip it captures the pointer, builds a position:fixed faded clone that follows the cursor, drops a card-sized placeholder where the card sits, and dims the original. On pointermove it finds the nearest other card by 2D center-distance and decides before/after by the dominant axis (vertical gap → compare Y; horizontal → compare X), then moves the placeholder there — that placeholder is the visible landing-slot highlight. On pointerup it counts real cards before the placeholder to get the target index and calls onDrop(targetIndex); Escape or pointercancel aborts. Because it uses pointer events, it works with a mouse and with touch.
  • The highlighted landing slot. New CSS .drop-placeholder — a dashed accent border with a soft accent fill and inset glow — is the slot that shows where the card will land. .drag-clone is the fixed, slightly-scaled, shadowed copy that trails the cursor (z-index below the modals); .card-dragging dims the source. The old .drop-before/.drop-after indicator lines are kept because the bookmark rows still use them.
  • Hover-revealed ▲/▼ move arrows. A new helper moveCardInOrder(orderArray, key, direction, keyOf) (supporting earlier | later | front | end) plus buildMoveControls add up/down arrows between the grip and the title on every widget and category header. They’re hidden until you hover the card, and clicking one nudges the card one spot and saves through setState. (front/end exist in the helper but aren’t surfaced as buttons, to keep the headers clean.)
  • Correct index math, traced by hand. Widget drops do an anchor-key splice of persona.widgetOrder so hidden widgets don’t throw off the result; category drops splice persona.categories at the target index. The Build Agent traced both — the card lands exactly where the placeholder showed.
  • Scope & verification. Only index.html changed. node --check passes. The native drag for bookmark rows, persona tabs, and the config lists (tickers, weather cities, clock zones) is deliberately untouched and still works. A live in-browser test wasn’t possible in the sandbox (the app needs gist auth to fully load), so the Build Agent recommends Tom smoke-test.

Decisions made

  • Custom vanilla pointer-drag, no library. The native HTML5 drag was unreliable and, after v1.15’s masonry change, guessed the wrong axis (it still compared cards as a horizontal row). One shared pointer-based engine fixes both, adds touch support for free, and avoids pulling in any dependency.
  • Before/after by dominant axis. Comparing on whichever axis has the larger gap to the nearest card makes the landing slot correct whether cards are stacked vertically (masonry) or sitting side-by-side.
  • Widget index by anchor key, not raw position. Splicing widgetOrder relative to the dragged card’s neighbor key means hidden widgets in the order array don’t shift the drop result.
  • Arrows surface earlier/later only. “To front” / “to end” exist in the helper but aren’t buttoned, to keep card headers uncluttered — a possible future polish, along with extending the new pointer engine to the bookmark/tab/config-list drags.

v1.13 — Ask Gemini chat resilience: it now auto-retries Google’s transient “model overloaded” 503s instead of surfacing them, with a Retry button if retries run out. Also closed the v1.11 Gemini CORS open question — Tom confirmed the chat works in his browser. Real Build Agent + Documentation Agent subagent flow.

Completed

  • Auto-retry on Google’s “overloaded” 503. A new chatIsOverload(res) check treats a non-OK reply as overloaded if it’s HTTP 503 or its error text contains “overloaded”/“high demand” (case-insensitive). When that happens the chat waits a short backoff (CHAT_OVERLOAD_BACKOFFS = [1500, 3000] ms plus up to 500ms random jitter) and re-issues the same request, up to CHAT_OVERLOAD_MAX_RETRIES = 2 (so 1 initial try + up to 2 retries = 3 total). Most transient blips clear on the first retry.
  • No error flashes mid-retry. The “Gemini is thinking…” indicator is created once before the retry loop and removed only at a terminal outcome, so it stays up across retries — the user never sees an interim error.
  • Plain-English give-up note + Retry button. If all retries fail, a new appendOverloadNote() shows a dashed note bubble: “Google’s servers are briefly overloaded — this is on Google’s side, not you. Try again in a moment,” with an “Open full Gemini ↗” link-out and a Retry button. Retry re-runs the request directly (it removes its own note first, doesn’t re-add the user’s message, and is guarded by chatSending). New CSS: .chat-msg.note .chat-retry (+ :disabled), styled like the send button from theme tokens — light and dark safe.
  • Internal refactor to make Retry clean. sendChatMessage was split into sendChatMessage (validates, pushes the user turn to history exactly once) and runChatRequest (builds and issues the API call with the last ~10 turns and holds the retry loop). This lets the Retry button reuse the request without duplicating the user’s message.
  • Everything else unchanged. 429 rate-limit handling (no retry) and network/CORS handling are untouched. No new dependencies, no new keys; only index.html changed (chat CSS + chat JS).

Decisions made

  • Short auto-retry, not long exponential backoff. A chat needs to feel responsive, so retries are capped at 2 with ~1.5s/3s waits rather than 30s+ exponential backoff.
  • No automatic model-switching fallback. The default gemini-2.5-flash tier is already relatively stable; auto-retry on the same model is the targeted fix, and swapping models is out of scope.
  • 503 is distinct from 429. A 503 (Google overloaded) auto-retries; a 429 (your rate limit) does not — retrying a rate limit just makes it worse.

Resolved

  • v1.11 open question closed: the Gemini browser-direct call works. Tom confirmed the Ask Gemini chat works in his browser — the native :generateContent endpoint is not CORS-blocked. That open item is now removed from the Open Items list. (The graceful link-out fallback remains for any future connection hiccup.)

Why: Twelve Data’s free tier caps at 8 price lookups per minute, and Tom’s ~12 tracked symbols routinely blew past that — producing the constant HTTP 429 (“too many requests”) errors he was seeing. Finnhub’s free tier allows 60 calls a minute (no credit card). The old fetchTwelveDataQuotes was removed and replaced with fetchFinnhubQuotes(symbols), which makes one /quote call per symbol (fanned out in parallel) and normalizes Finnhub’s c/d/dp fields into the dashboard’s existing price/dollar/percent shape. If one symbol fails it shows “—” rather than killing the whole widget. Symbol search moved to Finnhub’s /search endpoint; the ETF proxies (SPY/QQQ/DIA) still appear. The Settings field, its inline walkthrough, and all empty-state messages now point at Finnhub (finnhub.io/register). The old Twelve Data key is still read for backward-compat but is no longer used by anything.
  • Shared 5-minute price cache. Replaced the old per-tab in-memory quote cache with a localStorage-backed cache (sd_quote_cache, 5-minute TTL) that every tab and reload draws from. Only stale or missing symbols hit the network — opening the dashboard in three tabs no longer triggers three rounds of API calls. Helpers readQuoteCache() / writeQuoteCache() manage it.
  • Smarter rate-limit handling. If every symbol is rate-limited with nothing cached, the widget shows “Rate limit reached — prices refresh shortly” and schedules a single refresh 60 seconds later. The v1.8 immediate 2-second retry was dropped for stocks — it actually made rate-limiting worse. (Crypto / CoinGecko keeps its v1.8 single retry and was otherwise left alone.)
  • Drag-to-reorder overhaul with visible grip handles. The old HTML5 drag was undiscoverable — nothing showed Tom what to grab. Now categories, bookmarks, and every config list (stock tickers, weather cities, clock zones) prepend a draggable ⠿ grip handle (a grab cursor, brightens on hover). A blue drop-indicator line shows where the item will land. Bookmark drops are position-aware (drop on the top or bottom half of a row to insert before/after) via a new moveBookmark() helper that handles both within-category and cross-category moves; dropping on empty category space still appends. Config-list reordering is handled by a new shared makeCfgItemDraggable() helper.

Decisions made

  • Finnhub over Twelve Data — 60 calls/min vs 8 on the free tier. Tom’s symbol count made Twelve Data’s limit a constant source of 429s; Finnhub clears that headroom. Both are free, so no cost change.
  • Shared localStorage quote cache, not per-tab in-memory — multiple tabs and reloads were multiplying API calls. A 5-minute cache shared across tabs cuts that down and keeps Tom comfortably under the rate limit.
  • On 429, back off 60 seconds instead of retrying immediately — v1.8’s 2-second retry made rate-limiting worse by piling on more calls. One delayed refresh is gentler and recovers cleanly.
  • Visible ⠿ grip handles — bare HTML5 drag had no affordance; Tom literally couldn’t find anything to grab. An explicit grip makes the drag target obvious.
  • Per-symbol graceful degradation — dropped the stock widget’s cold-start retry; a single failed symbol now shows “—” instead of taking down the widget. Skipped adding a localStorage cache for crypto to keep this change focused.
